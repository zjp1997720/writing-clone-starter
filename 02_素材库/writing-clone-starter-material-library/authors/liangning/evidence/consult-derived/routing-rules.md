# 咨询天团路由规则

主持人做路由时，**先读 manifest，再套规则**。

唯一 roster 来源：

- `references/advisor-manifest.json`

只有同时满足以下条件的顾问，才算当前可调用：

- `validation_status = "passed"`
- agent 文件存在
- 如果是 discuss 模式，`discuss_eligible = true`

---

## 第一步：先判模式（快速咨询 vs discuss）

### 默认留在快速咨询

- 用户要的是建议、框架、清单、优先级，顾问之间不需要当场互相追问。
- 多位顾问只是补充视角，适合并行回答后由主持人整合。
- 问题虽然复杂、战略级、跨领域，但主要是信息拼接，不靠正面碰撞也能给出可执行建议。
- 用户明确要快答、要多角度扫描、要全员快速看法。

### 升级到 discuss 圆桌

- 用户明确要求 `--discuss`，或明确想看某几位专家互相辩论。
- 这个决策的关键，不在于多找几个人，而在于让不同立场正面交锋后再收敛。
- 预期存在真实分歧：路径选择互斥、先后顺序冲突、目标函数不同、同一事实有竞争性解释。
- 只有通过追问、反驳、让对方回应，才能看清代价、边界和取舍。

### 明确不要滥用 discuss

- 不能只因为问题“很大 / 很重要 / 很复杂 / 涉及 3 个以上领域”就自动开 discuss。
- 不能只因为想听全员意见就开 discuss；这类需求优先用全员快速咨询。
- 不能把“上下文还不够”当成 discuss 理由；先补上下文，再决定是否需要碰撞。
- 如果主持人已经能预判这些顾问大概率只是互补，不会形成有价值的分歧，就留在快速咨询。

---

## 第二步：再判顾问（manifest 驱动）

### 路由输入

主持人必须把问题拆成两层：

1. **显性关键词**
   - 问题里直接出现的行业词、动作词、平台词、方法词
2. **隐性目标函数**
   - 用户到底在要什么：增长、定位、变现、用户理解、长期方向、执行突破

### 路由方法

对每个已注册 advisor，用 manifest 中三组字段做匹配：

- `routing_keywords`：直接词面命中
- `routing_tags`：语义层分类命中
- `core_domains`：最终解释和展示给用户的能力摘要

推荐做法：

- 词面强命中 + 标签命中 → 优先候选
- 只有词面命中但目标函数不一致 → 降权
- 词面不明显，但目标函数高度匹配 → 可补位入选

### 选人数量原则

- 单一领域问题 → 1 人
- 双领域交叉问题 → 2 人
- 多维度但仍以补充视角为主 → 2-3 人
- 用户明确 `--all` → 所有 `validation_status=passed` 的顾问
- discuss 模式 → 2-4 人，优先挑会产生真实分歧的组合

## 组合推荐原则

主持人不要死记组合表，而要按“目标函数是否冲突”选人。

### 适合快速咨询的组合

- 互补型组合：一个人看战略，一个人看执行，一个人看商业闭环
- 扫描型组合：用户想快速知道不同角度有没有明显盲区

### 适合 discuss 的组合

- 对同一问题，判断标准天然不同
- 至少一方会质疑另一方的目标函数、优先级或时机判断

### 当前高分歧示例

| 组合           | 典型分歧点                           |
| -------------- | ------------------------------------ |
| 刘润 vs 润宇   | 交易结构是否成立 vs 模式是否已经失效 |
| 薛辉 vs 润宇   | 流量优先还是产业优先                 |
| 罗振宇 vs 薛辉 | 长期主义 vs 快速变现                 |
| 梁宁 vs 刘润   | 用户需求优先 vs 商业效率优先         |
| 梁宁 vs 薛辉   | 产品打磨 vs 快速测试                 |

## 路由决策优先级

1. **用户指定的模式和人选 > 一切规则**
2. **manifest 可调用性 > 路由偏好**（未通过校验的顾问不能入选）
3. 明确需要碰撞、反驳、校正彼此假设 → discuss，选 2-4 位最可能产生有效分歧的专家
4. 单一领域问题 → 快速咨询，对应单人
5. 跨 2 个领域的问题 → 快速咨询，对应 2 人
6. 跨多个领域但主要是补充视角，不靠争论出结论 → 快速咨询，选 2-3 人，必要时 `--all`
7. 顾问人选拿不准时，宁多勿少；**模式拿不准时，先留在快速咨询**

---

## discuss 模式补充

圆桌讨论至少需要 2 位专家。选人时优先选**可能产生分歧**的组合；如果只是想补充更多视角而不是制造对质，不要切 discuss。

主持人组桌时必须再检查一遍：

- 每位参与者 `discuss_eligible = true`
- 每位参与者都有稳定的 `artifact_name`
- 每位参与者的 agent 文件存在且可读取
