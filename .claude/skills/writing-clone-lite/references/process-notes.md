# Writing Clone Lite - Process Notes

这份文件沉淀轻量写作场景里已经验证过的偏好。

它的职责是：

- 展开 Lite 已经确定的执行偏好
- 细化形式层返工顺序与复盘信号
- 帮助主代理在不滑回重型流程的前提下把正文写稳

它**不是**第二份主 skill，也不负责改写 Lite 的核心合同。

如果这份文件与 `SKILL.md` 的表达层面出现差异：

**以 `SKILL.md` 为准。**

## 当前已确认的写作偏好

- 默认优先规整而高密度
- 默认对齐高信息密度、完整自然段优先的大鹏长文风格
- 正文优先写成 prose-first、散文化、弱列表感、成片推进的版本
- 标题允许少量分层，但默认禁止数字序号标题
- 短句只用于提气、转折、收口，不全篇碎切
- hook / 转折 / 收束可以单独成段；正文中段优先用完整自然段承载机制和判断
- 如果出现连续 3 个以上超短正文段，默认说明写碎了，要主动并段
- bullet 只留给步骤、对比、清单，不让无序列表承载正文主体
- 结构要藏在文气里，不写成 PPT 提纲感
- 非说话场景的中文双引号 `“ ”` 统一替换成 `「」`
- 当当前对话已经形成充足上下文时，不重复执行重型上下文采集
- 终稿默认对齐公众号消费协议：`title / summary / author`
- 当任务明确是“公众号头条 / 推荐流放大”时，先检查标题和封面是否在表达“读者即时可带走的价值单元”；如果入口更像概念命名、产品命名或诊断式表达，优先重写入口，不要急着继续打磨正文

## 新增：规整高密度版硬门槛

下面这些不是建议，是 Lite 默认必须守住的形式门。

### 1. 正文主体禁碎

默认目标不是“有节奏感”，而是：

**规整 + 成片 + 高密度。**

节奏感只能建立在规整基础上，不能靠满篇碎段硬制造。

### 2. 短句只能局部提气

单句成段默认只允许出现在：

- 开头 hook
- 段间转折
- 结尾收束

正文中段，如果一个判断本来应该用完整自然段承载，就不要拆成很多小锤子句子。

补一句更重要的：

**先承载，再提气。**

也就是说，短句不是正文推进策略。

短句是局部增压器。

如果一个小节的主判断还没站稳，就不要急着靠短句制造气口。

### 2.5 规整底盘优先级

Lite 的默认优先级应该是：

1. 先让正文主判断站稳
2. 再让段落之间更顺
3. 最后才决定哪里需要短句提气

如果“规整”和“节奏感”冲突，默认先保规整。

如果“完整承载”和“金句感”冲突，默认先保承载。

### 2.6 短句预算（仅作复盘信号，不是硬配额）

这不是生成时的硬限制。

这是写后复盘时的告警信号。

用于提醒你有没有把短句从“提气工具”用成“主推进方式”。

默认检查：

- 一个主要小节里，短句通常只应占局部点缀位置
- 如果一个小节里短句多到开始承担解释功能，说明已经越权
- 如果删掉这些提气短句后，正文主线反而更顺，说明短句使用过量

不要把“短句预算”理解成数字配额。

它只是提醒你：**短句有没有抢走完整自然段的活。**

### 3. 每个主要小节至少有一个完整承载段

二级标题下面，至少要有一个能完整承载主判断、机制解释或关键转折的自然段。

不能全靠：

- 单句
- 双句小段
- bullet
- 零碎判断堆砌

### 4. bullet 不能承载正文主逻辑

无序列表只允许用于：

- 步骤
- 对比
- 清单
- 明确枚举项

如果删掉 bullet 后改写成自然段，文章会更顺，就优先删 bullet。

### 5. 连续碎段触发返工

如果正文连续出现以下任一情况，默认判定写碎了：

- 连续 3 个以上超短正文段
- 连续 2 个以上单句短段
- 一个小节里几乎全是短段，没有承载段

一旦命中，不允许直接交稿，必须先做并段重写。

返工顺序默认是：

1. 先补完整承载段
2. 再压掉多余碎段
3. 最后保留真正有用的提气短句

### 6. 讲义感 / 提纲感也算失败

如果正文读起来像：

- 研究汇报
- PPT 提纲
- 朋友圈拆句长文
- 视频口播切句稿

即使信息是对的，也算形式失败。

## 使用提醒

这份 process notes 不是风格 profile。
它更像 Lite 执行偏好的备忘录。

如果当前 profile 下已有 `form-normalized-anchors/`，这份文件默认和 normalized form 一起构成 Lite 的运行时执法层：

- normalized 决定“像什么样的长文”
- process notes 负责把这些 form 规则真正卡住

如果用户明确反馈：

- 「太散了」
- 「太像讲义」
- 「列表太多」
- 「句子太碎」
- 「不够工整」

就优先回来看这份文件，而不是第一时间去补更多素材。
