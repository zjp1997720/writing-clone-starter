# Goldmine 动态检索策略

目标：让 `writing-clone-profile` 调用素材库时，优先依赖**动态检索**，而不是依赖人工维护的静态导航表或固定目录记忆。

---

## 1. 核心原则

### 1.1 动态检索优先，静态导航降级

优先做：

- 根据当前 Brief 动态生成查询词
- 在 `02_素材库/` 下按相关性召回
- 按当前文章缺口决定要补哪个槽位

不要做：

- 先翻固定导航文档再决定写法
- 把某个作者/某个目录写死为默认入口
- 因为目录结构变化就丢失素材调用能力

### 1.2 素材库是增强器，不是主笔

优先级始终保持：

1. 用户本轮对话与 Brief
2. 当前 Profile 的 `style-profile.md` / `form-normalized-anchors/` / `voice-cues.md`（如果存在）
3. 当前 Profile 的 `examples.md` / `quotes.md`
4. memory / repo 内相关上下文
5. Goldmine 素材库

只要 Goldmine 命中项与当前 Profile 的风格、真实经历、文章立场冲突，直接丢弃，不做折中缝合。

### 1.3 先定槽位，再搜素材

Goldmine 不应该先搜一堆再硬塞进文章。

正确顺序：

1. 先判断当前文章缺什么：
   - `hook`
   - `structure`
   - `gold_sentence`
   - `ending`
2. 再为这个槽位生成查询词
3. 再从结果里挑 1-2 条最能补这个槽位的素材

---

## 2. 何时启用 Goldmine

满足任一条即可启用：

1. 当前 Brief 缺明确 hook
2. 当前 Brief 缺结构骨架
3. 当前 Brief 需要更高的金句密度
4. 当前主题偏方法论/框架型，适合借结构而不是借故事

满足任一条必须跳过：

1. 当前文章强依赖个人经历、个人情绪、个人判断
2. 当前 Profile 已经提供了足够强的表达和案例
3. 检索结果相关度不够，或无法明确说明要放在哪个槽位
4. Goldmine 素材一旦加入就会明显冲淡「大鹏在说话」的感觉

---

## 3. 默认调用强度

### 低使用：0-2 条

适用：

- 强个人经历文
- 决策复盘文
- 高情绪/高顿悟文章

只补：

- 1 条 hook 或
- 1 条 ending

### 中使用：2-4 条（`method / framework` 默认）

适用：

- 方法论拆解文
- 工具教程文
- 框架型公众号长文

建议分配：

- hook：1 条
- structure：1 条
- gold_sentence：1-2 条

### 高使用：4-6 条

只适用：

- 纯框架型文章
- 非个人叙事驱动内容

即便启用高使用，也只能借：

- 模式
- 结构
- 节奏

不能直接拼原句、拼段落。

## 4. 动态检索流程

开始检索前，先写一张 `Goldmine Budget Card`：

- `article_mode`: `personal / judgment / method / framework`
- `slot_gap_map`: 当前真正缺的是哪个槽位
- `goldmine_usage_mode`: `skip / low / medium / high`
- `query_terms`: topic / slot / stance 三组查询词
- `hard_budget`: 总条数、单槽位上限、单条摘录上限
- `drop_rules`: 哪些命中会压过作者本人，因此必须丢弃

强制规则：

- `personal / judgment` 默认 `skip` 或 `low`
- `method` 默认 `medium`
- `framework` 才允许视情况升到 `high`

### Step 1：先抽取检索输入

从 Brief 里提取：

- `topic`
- `stance`
- `target_reader`
- `must_include`
- `must_avoid`
- 当前缺口槽位：`hook / structure / gold_sentence / ending`

### Step 2：生成查询词

每次至少准备 3 组查询：

1. **主题查询**：主题词 + 目标读者痛点
2. **槽位查询**：主题词 + 当前槽位
3. **立场查询**：主题词 + 核心立场 / 反直觉判断

示例：

- `AI 写作 活人感 hook`
- `一人公司 内容系统 structure`
- `上下文工程 不是提示词 gold_sentence`

### Step 3：优先用 Obsidian / vault 检索

优先搜索根目录：

- `02_素材库/`

不要一开始就把范围写死到某个子目录；子目录只是结果过滤条件，不是入口假设。

推荐顺序：

1. 在 `02_素材库/` 全局检索主题 + 槽位词
2. 如果结果太多，再按类型收缩到：
   - `02_素材库/灵感素材/标题/`
   - `02_素材库/灵感素材/开头/`
   - `02_素材库/灵感素材/金句/`
   - `02_素材库/灵感素材/结构/`
3. 如果结果太少，再补：
   - 作者名
   - source_label
   - 最近新增 / 最近修改

### Step 4：给结果打分

每条候选至少从这 4 个维度打分：

1. **主题相关度**（0-5）
2. **槽位适配度**（0-5）
3. **风格兼容度**（0-5）
4. **可复用性**（0-5）

只有满足下面条件的结果才保留：

- 综合判断 >= 4
- 能明确说明用于哪个槽位
- 不会压过当前 Profile 的说话方式

### Step 5：只带少量结果进入 Context Packet

硬预算：

- 总条数 <= 6
- 默认优先 2-4 条
- 单条摘录 <= 200 字
- 每个槽位默认最多 1 条（gold_sentence 最多 2 条）

记录格式：

- `source_type: library`
- `source_path_or_id`
- `content`
- `relevance`
- `notes`
- `usage_slot`

---

## 5. 目录变化时怎么处理

如果素材库目录结构变化：

1. **不要先改导航文档**
2. 先把检索根目录保持在 `02_素材库/`
3. 依赖关键词、正文、frontmatter、作者标签去找
4. 只有在结果明显失真时，才补新的过滤规则

换句话说：

- 目录是辅助信息
- 可检索文本才是主信息

---

## 6. 允许的使用方式

优先借：

- 结构骨架
- 开头模式
- 金句节奏
- 结尾收束方式

谨慎借：

- 原句本身
- 过强的人设表达
- 带明显作者烙印的判断句

禁止：

- 整段拼贴
- 一篇文章里连续塞多个外部作者味道
- 为了显得有料而牺牲活人感

---

## 7. 未来优化方向

如果后面继续升级素材库，优先补这几类可检索元信息：

- `usage_slot`
- `source_author`
- `topic_tags`
- `stance_tags`
- `style_tags`

这样动态检索会更稳，但在这些字段还没补齐之前，也应先靠全文检索 + 槽位判断工作，而不是退回静态导航。
