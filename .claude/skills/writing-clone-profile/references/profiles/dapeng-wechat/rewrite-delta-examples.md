# Rewrite Delta Examples（dapeng-wechat）

这份文件记录真实 `ai-baseline -> final` 或人工 diff 里反复出现的改稿差异。

它不是让模型背诵 final 句子，而是让模型学会：大鹏把 AI 稿改成人味稿时，通常改的是哪一层。

## 1) 使用边界

- 优先用于二稿修订、去 AI 味、Human Feel Review 失败后的模块重写。
- 不默认把这里的例子当开头模板直接套用。
- 只抽 delta 模式，不迁移一次性事实、数字、工具版本和截图链接。
- 如果只有 baseline 没有 final，不写成「大鹏最终会这样改」，只能标成 anti-pattern。

## 2) Delta A：抽象体验 -> 动作 + 机制 + 结果

来源：

- `01_项目/内容创作/260402_个人上下文资产/03-Reviews/人工改稿-diff复盘.md`

baseline 问题：

- 只说「完成得很好」「效果不错」。
- 读者不知道作者做了什么、AI 靠什么完成、结果具体长什么样。

final 倾向：

- 补齐三层：动作、机制、结果。
- 例如把「整理得很好」推进到「把录音文件交给工具、利用语义召回、很快产出需求与功能对照」这一类可想象结果。

写作规则：

- 关键 aha moment 不许只写抽象感受。
- 至少补一层动作和一层结果；如果知道机制，再补机制。

不适用：

- 没有真实发生过的项目结果。
- 没有来源的数据或时间。

## 3) Delta B：资料说明 -> 个人现场

来源：

- `01_项目/内容创作/260411_AI时代你还在不在场/03-Reviews/草稿_final.ai-baseline.md`
- `01_项目/内容创作/260411_AI时代你还在不在场/草稿_final.md`

baseline 问题：

- 观点成立，但像整理过的会议感想。
- 作者在场感不够强，读者看不到「我当时在哪、看到什么、为什么被击中」。

final 倾向：

- 加入现场截图、具体时间、真实工具名和更贴近当下的自我矛盾。
- 把抽象焦虑落到「我好像很久没主动想过新问题」这一类自我观察。

写作规则：

- 写会议、活动、课程、对话类文章时，先问：有没有一个真实现场能把观点钉住？
- 能用具体动作就不要只写抽象情绪。

不适用：

- 为了显得真实而硬造现场。
- 公开内容不适合暴露的私密场景。

## 4) Delta C：通用方法论 -> 当前真实栈

来源：

- `01_项目/内容创作/260413_爆改栋哥dbs-skill/03-Reviews/草稿_final.ai-baseline.md`
- `01_项目/内容创作/260413_爆改栋哥dbs-skill/草稿_final.md`

baseline 问题：

- 能解释方法论 Skill 的价值，但读起来像在介绍一个类别。
- 用户不知道大鹏到底怎么改、改完有什么差异。

final 倾向：

- 直接写「爆改」后的真实结果。
- 把 `nmem`、`Obsidian CLI`、课程份数、激活率、定价等真实上下文接进去。
- 让 Skill 的价值从「它很聪明」变成「它拿着我的历史数据来问我」。

写作规则：

- 写工具/方法论时，优先把抽象能力接到自己的真实工作栈。
- 能写「它现在会先问我上次承诺做没做」就不要只写「它更懂我」。

不适用：

- 工具栈还不稳定、事实没核验时。
- 为了具体而堆配置细节，导致读者忘了主线。

## 5) Delta D：全景展示 -> 聚焦实战碰撞

来源：

- `01_项目/内容创作/260416_咨询天团实战效果/03-Reviews/草稿_final.ai-baseline.md`
- `01_项目/内容创作/260416_咨询天团实战效果/草稿_final.md`

baseline 问题：

- 人物多、信息多，但容易变成「大家都很厉害」。
- 读者看到的是配置，不是碰撞。

final 倾向：

- 聚焦少数关键角色和真实交锋。
- 用具体争论、具体追问、具体决策展示价值。
- 结论回到：圆桌的价值不是多几个答案，而是逼出盲区。

写作规则：

- 多角色、多案例、多功能文章，不要平均分配篇幅。
- 选 2-4 个最有张力的点，把剩下的压成背景。

不适用：

- 教程文章需要完整覆盖步骤时。

## 6) Delta E：复盘叙述 -> 数据化、关系化、可执行

来源：

- `01_项目/内容创作/260417_AI自媒体变现复盘/03-Reviews/草稿_final.ai-baseline.md`
- `01_项目/内容创作/260417_AI自媒体变现复盘/草稿_final.md`

baseline 问题：

- 复盘顺序成立，但数据和关系还不够硬。
- 有些判断像经验总结，不像用真金白银换来的教训。

final 倾向：

- 补具体阅读、转发、完读、收入、销售份数等数据。
- 把「流量下降」和「营销意图前置」建立清楚因果关系。
- 给出渠道分工：公众号建信任，视频号塑人格，直播收钱。

写作规则：

- 经营复盘要回答：哪个动作导致了什么后果。
- 建议必须从前文的坑里长出来。

不适用：

- 没有可靠数据时，宁可写主观复盘，不要编精确数字。

## 7) Delta F：抽象结尾 -> 再推进半步

来源：

- `01_项目/内容创作/260402_个人上下文资产/03-Reviews/人工改稿-diff复盘.md`
- `01_项目/内容创作/260411_AI时代你还在不在场/草稿_final.md`

baseline 问题：

- 结尾已经正确，但停在概念总结。

final 倾向：

- 往一个更可想象的终局画面或个人规矩推进半步。
- 不是再总结一遍，而是让读者看到「接下来该怎么活 / 怎么做」。

写作规则：

- 结尾问：这句话之后，读者能不能多一个动作、多一个画面或多一个判断？
- 如果不能，结尾还差半步。

不适用：

- 事实密集教程，结尾应优先清晰，不强行诗化。

## 8) 快速修稿清单

当一篇草稿 AI 味重，按顺序看这 6 个 delta：

1. 有没有关键案例只写了「效果很好」？
2. 有没有观点成立但作者不在场？
3. 有没有通用方法论没有接到真实工具栈、业务数据或项目动作？
4. 有没有多案例 / 多人物平均用力，缺少碰撞和递进？
5. 有没有复盘只讲经过，没有讲动作与后果的关系？
6. 结尾有没有停在正确总结，没有再推进半步？
