# WeChat to Teleprompter

把一篇已经成型的公众号稿，稳定改成一个**能直接录视频号**的提词稿。

它不是短视频运营顾问。
它也不是文章摘要器。

它只解决一件事：

**当内容已经成熟，怎么把它改成提词器友好、嘴上能念、镜头里眼神更稳的版本。**

---

## 什么时候用

命中任一场景就用：

- 用户说「把这篇公众号改成视频号口播稿」
- 用户说「改成提词稿」「提词器版本」
- 用户已经给出 `草稿_final.md`、公众号终稿 Markdown、长文 Markdown
- 用户直接粘贴了一整篇正文，目标是尽快进入录制
- 当前任务的瓶颈是“转换形态”，不是“补上下文”

典型表达包括：

- 「帮我改成视频号口播稿」
- 「做成能对着念的版本」
- 「做个提词器友好的版本」
- 「内容不用改方向了，只优化录制」

---

## 什么时候不要用

遇到下面这些情况，不要硬上我：

1. 用户只有一个模糊选题
2. 用户没有成型文章，只有零散想法
3. 当前最大的瓶颈还是“选题 / 立场 / 补素材”
4. 用户要的是账号诊断、变现咨询、爆款策略
5. 用户要从 0 写一篇文章，而不是把现有文章转成视频号提词稿

判断标准很简单：

**如果现在最大的瓶颈还不是“提词录制”，就不要用我。**

回退口径：

`这轮更像“先把内容想清楚/写成文”，不是“把成稿转成提词稿”。这时候不要用 wechat-to-teleprompter，应该切到写作类或短视频策略类 skill。`

---

## Teleprompter Gate

正式转换前，先检查四项：

- `topic`：主题是否明确
- `thesis`：核心判断是否明确
- `body`：正文是否已经有完整论证材料
- `goal`：目标是否明确为视频号录制 / 提词

规则：

- 命中 3 项及以上：直接转换
- 命中 2 项：允许做一次极简确认，再继续
- 命中 1 项及以下：停止，回退

极简确认只允许问高杠杆问题，例如：

- 「这次更偏完整判断版，还是直接偏录制提词版？」
- 「你这篇是想保留大部分信息，还是主动压缩到更短版本？」

不要进入长轮追问。

---

## 输入协议

默认支持两种输入：

### 1. 文件输入

例如：

- `草稿_final.md`
- 任意公众号终稿 Markdown
- 任意成型长文 Markdown

优先读取：

- 标题
- summary（如果有）
- 正文主线
- 核心判断
- 必须保留的信息块

### 2. 正文输入

用户直接粘贴整篇正文时：

- 先识别主题与核心判断
- 如果有标题，用标题命名输出
- 如果没有标题，用主题生成一个最小可用文件名

不要因为输入不是文件，就拒绝转换。

---

## 默认输出

默认只输出一个终点文件：**提词版**。

这是用户真正消费的版本，不默认交付完整判断版和录制版。
提词版里默认附带 **5 个推荐短标题**，方便用户在发视频号时直接选。

建议命名：

- 输入为 `草稿_final.md` → 输出 `草稿_final.提词版.md`
- 输入为正文 → 在当前任务目录下创建同类命名文件

默认要求：

1. 不覆盖原文
2. 输出到原文同目录优先
3. 提词版必须能直接用于屏幕提词或提词器软件
4. 提词版默认包含 `5` 个推荐短标题

---

## 核心转换目标

提词版必须同时满足四个目标：

1. **关键信息不丢**
2. **嘴上念得顺**
3. **保留大鹏判断感**
4. **横向铺开，减少视线垂直跳动**

注意：

- 不要为了“短视频感”强删关键判断
- 不要把任务做成“摘要器”
- 不要把公众号原文直接碎切成一行一行的短句

如果原文明显属于这类文章：

- 误区拆解
- 认知升级
- 非共识判断
- 旧框架替换为新框架

转换时要额外保住它的认知骨架：

1. 旧印象 / 旧因果
2. 反例 / 现实冲突
3. 新框架
4. 关键概念定义与展开

这条规则的定位是：

- 它是**结构保真规则**，不是新增选题任务
- 只有原文已经具备这类骨架时才触发
- 不允许为了“做成反常识”去篡改原文立场

---

## 执行步骤

### Step 1：读取原文并抽取骨架

先压出这几项：

- `topic`
- `thesis`
- `must_keep`
- `compressible_parts`
- `recording_goal`

如果原文明显是“误区拆解 / 认知升级 / 非共识判断”型文章，再额外识别：

- `old_assumption`
- `counterexample_or_conflict`
- `new_frame`
- `key_definitions`

其中最关键的是两件事：

1. 哪些信息绝对不能丢
2. 哪些地方只是文章解释感太强，可以重组而不是照搬

### Step 2：重组为视频号口播顺序

默认目标不是“改短”。

默认目标是：

**把文章顺序改成更适合开口讲的顺序。**

优先顺序通常是：

`结论 / 问题钩子 -> 核心判断 -> 关键展开 -> 分人判断 -> 收口 CTA`

如果原文更适合“误区拆解”或“判断递进”，可以变体，但必须保留：

- 开头抓手
- 核心判断
- 必须保留的信息块
- 结尾互动或收口

对于认知型文章，优先保住这种变体顺序：

`旧印象 / 误区 -> 反例 / 现实冲突 -> 新框架 -> 关键概念定义与展开 -> 收口 CTA`

不要只留下一个反常识句子就往下走。
真正值钱的不是“反常识一句话”，而是把旧认知替换掉的完整过程。

### Step 3：做口播顺嘴优化

这一层只处理“嘴上能不能过”。

默认使用这些口播规则：

- 断长句
- 去书面词
- 去念不出来的符号
- 砍冗余修饰
- 并列不超过三项
- 用自问自答拆掉太长的干货段

参考：

- `references/style-notes.md`

### Step 4：输出提词器友好版

这是最终交付层。

重点不是再改内容，而是改排版。

要求：

- 尽量横向铺开
- 减少“一行一个短句”
- 让用户读的时候主要左右移动，而不是上下找句子
- 章节标题保留，方便快速定位
- 正文段落要兼顾屏幕阅读和嘴上节奏

### Step 5：补充发布层短标题

在提词稿定稿后，默认补 `5` 个推荐短标题。

规则：

- 标题要短，适合视频号发布栏位
- 必须围绕正文核心判断
- 不要标题党
- 不要脱离正文硬造冲突
- 5 个标题之间要有轻微差异，不要只是机械换词

默认放在最终文件的这个顺序里：

1. 提词稿正文
2. 推荐短标题
3. 视频简介
4. 标签

---

## 绝对边界

1. **不要覆盖原文文件**
2. **不要把任务做成摘要器**
3. **不要为了更短删掉关键判断**
4. **不要强行套薛辉式毒舌人格**
5. **不要默认进入账号诊断、变现咨询、选题生成**
6. **不要把排版重新做回一行一个短句的低空间利用模式**
7. **不要为了凑标题数量写出跑偏或标题党的短标题**

---

## 风格来源

这个 skill 可以借 `xuehui-video-ops` 的方法理解，但不继承它的人格系统。

允许借的只有：

- 开头先给结论或问题钩子
- 口播稿是写给嘴念的
- 断长句、去书面词、减冗余这些口播适配规则

默认语气：

- 贴近大鹏判断感
- 保留第一人称判断
- 保留信息密度
- 不进入重人格扮演

## 成功标准

完成后必须做到：

1. 用户拿到的是一个可直接录的视频号提词稿
2. 关键判断和核心信息块没有被过度压缩
3. 提词版读起来顺，屏幕里看起来稳
4. 原文保留不被覆盖
5. 默认附带 `5` 个可直接用于视频号发布的推荐短标题
6. 短标题与正文核心判断一致，不跑偏，不标题党

## 推荐输出口径

完成后回报：

```text
已生成视频号提词稿：<path>
- 输入来源：文件 / 正文
- 输出形式：提词版
- 处理重点：信息保留 / 顺嘴优化 / 提词排版 / 短标题补充
```
