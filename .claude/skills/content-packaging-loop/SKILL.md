# Content Packaging Loop

## 这是什么

这是一个给内容创作者、IP 型老板、一人公司创业者用的**内容包装闭环优化器**。

它不负责全文重写。
它负责的是另一层更高杠杆的东西：

- 标题
- 开头钩子
- 封面文案
- 切入角度
- 小标题结构
- 结尾 CTA

一句话。

**它不是再给你写一版，而是帮你把同一个内容，包装得更容易被点开、继续看、咨询和行动。**

---

## 什么时候用

命中任意两条，就优先考虑用我：

1. 用户明确说不要重写全文，只改包装层
2. 用户想要 3 个版本并从中选优
3. 任务目标和传播 / 转化有关，而不是单纯追求文采
4. 当前内容本体基本成立，问题主要卡在点开、抓停、行动
5. 用户想要一套可复用的迭代方法，而不是一次性灵感

典型表达包括：

- 「帮我优化这个标题」
- 「这篇别重写，只改开头」
- 「给我 3 个 CTA，选一个最适合长期信任关系的」
- 「围绕点击率继续优化」
- 「做成闭环，不要只给我几个版本就完了」

---

## 什么时候不要用

遇到下面这些情况，不要硬上我：

1. 用户要从 0 到 1 写整篇文章 / 整条口播 / 整套销售页
2. 用户还没想清楚主题、立场、读者和目标
3. 当前最大的瓶颈不是包装，而是内容本体过弱
4. 问题是品牌定位、商业模式、人生战略，而不是包装优化
5. 没有 baseline，也没有可执行的评测标准

判断标准很简单：

**如果现在最大的问题还是“这篇到底想说什么”，就别用我。先把内容本体长出来。**

---

## 核心原则

### 1. 先冻结 baseline，再谈优化
没有 baseline，就没有闭环。

### 2. 每轮只改少数变量
不要一轮同时重写标题、开头、CTA、结构、定位。那不是迭代，是推倒重来。

### 3. 评测先做成评分表，不先做成陪审团系统
默认由主 Agent 按 rubric 评分。Task / Sub Agent 只是增强模块，不是默认依赖。

### 4. 不用没人真的会设定的指标
不要引入 token 预算、奇怪模型分、抽象困惑分。只用创作者和老板真正关心的标准。

### 5. 不为了优化而优化
如果新版本没有明显更好，宁可保留 baseline，也不要硬改。

### 6. 沉淀经验比单次输出更重要
每轮必须总结一条可复用的包装经验。

### 7. 内容质量高于工具感
如果版本一眼看上去很像 AI 工业糖精，即使分数不差，也不要保留。

---

## 默认配置

- 候选版本数：3
- 默认迭代轮数：1-2 轮
- 每轮变量数：1-2 个
- 默认评测模式：主 Agent
- 可选增强评测：Task / Sub Agent
- 默认输出：baseline + 候选版本 + 评分 + 保留结论 + 本轮沉淀

---

## 输入合同

正式优化前，先尽量补齐这 6 项。  
如果当前对话里已经有，就不要重复追问。  
如果缺失，只追问最关键的 1-2 项。

### 1. 内容对象
例如：
- 公众号文章
- 视频号口播
- 小红书图文
- 朋友圈文案
- 私域引流文案
- landing page 第一屏

### 2. 目标用户
例如：
- 做 IP 的个人老板
- 一人公司创业者
- 已有业务闭环、开始做内容的老板
- 想做个人品牌但不会包装的人

### 3. 核心目标
必须具体，优先用真实世界目标：
- 更高点击率
- 更高停留
- 更高收藏率
- 更高咨询率
- 更高加微信率
- 更高回复率

### 4. 可编辑变量
每轮只允许 1-2 个变量进入实验，例如：
- 只改标题
- 标题 + 开头
- 开头 + CTA
- 封面标题 + 切入角度

### 5. 护栏
至少确认这些边界：
- 不标题党
- 不夸大承诺
- 不偏离原观点
- 不牺牲长期信任
- 不写出明显 AI 味

### 6. 预算
默认建议：
- 候选版本数：3
- 迭代轮数：1-2 轮
- 每轮改 1-2 个变量
- 没明显提升就停

## 执行流程

### Step 0：收紧问题定义
先把任务收紧成一句话：

> 这轮我要优化的不是全文，而是 [变量]；目标是让 [目标用户] 更容易产生 [动作]。

如果这句话说不出来，说明问题还没收紧，先别进入优化。

### Step 1：冻结 baseline
先把当前版本锁住。

至少记录：
- 当前标题 / 开头 / CTA / 封面文案（按任务相关项）
- 当前内容的核心观点
- 目标用户是谁
- 这次想提高什么结果

如果用户没有 baseline，但给了原文草稿，也可以从原文里抽出 baseline 包装层。

### Step 2：定义本轮实验
明确三件事：
- 本轮只改哪些变量
- 为什么优先改这些变量
- 本轮假设是什么

示例：
- 「本轮只改标题，因为内容本体已够强，传播差异主要卡在点开动作。」
- 「本轮只改前 3 句开头，因为用户会不会继续看，主要死在开场抓停。」

### Step 3：生成候选版本
默认生成 3 个版本，且每个版本必须策略差异明显。

建议差异方向：
- A：更强冲突
- B：更强利益点
- C：更强人群痛点

禁止给 3 个只是换同义词的伪版本。

### Step 4：主 Agent 默认评测
默认由主 Agent 对 baseline + A/B/C 做结构化评测。

#### 评分 rubric（每项 1-5 分）
1. **抓停力**  
   是否更容易让目标用户停下来继续看 / 点开。

2. **利益清晰度**  
   用户能否快速看懂自己能得到什么。

3. **人群贴合度**  
   是否真的像在对目标用户说话，而不是泛泛而谈。

4. **人设一致性**  
   是否符合用户既有风格、人设和长期信任资产。

5. **行动拉力**  
   是否更容易推动下一步动作，例如点击、停留、咨询、评论、加微信。

#### 护栏检查
每个版本都必须检查：
- 是否标题党
- 是否夸张失真
- 是否偏离原内容重点
- 是否显得廉价、套路、工业糖精
- 是否让真实人设失真

#### 结论规则
- **明显优于 baseline**：保留
- **分数接近，但更简洁、更稳**：可保留
- **有一点提升，但代价是更俗、更夸张**：丢弃
- **没明显提升**：保留 baseline，不硬选

### Step 5：可选 Task / Sub Agent 增强评测
仅在以下情况启用：
- 用户明确要求增加评审视角
- 主 Agent 难以在两个强版本之间做判断
- 需要模拟不同角色的视角
- 需要盲评，减少单一视角偏差

#### 这不是默认项
默认不要自动开启。只有用户明确要求，或你判断这轮值得额外评审时，才建议启用。

#### 推荐增强模式
**模式 A：目标用户视角评测**
- 子 Agent 扮演目标用户
- 重点看：是否会点开、是否会继续看、是否有痛感

**模式 B：编辑视角评测**
- 子 Agent 扮演内容编辑
- 重点看：表达是否稳、是否廉价、是否失真、是否长期可用

**模式 C：转化视角评测**
- 子 Agent 扮演经营者 / 增长视角
- 重点看：是否更容易推动咨询、留资、回复、行动

#### 输出要求
Task / Sub Agent 只做评审，不负责另起炉灶重写版本：
- 给出偏好排序
- 说明每版最强点和风险点
- 明确推荐保留谁

最终裁决仍由主 Agent 完成。

### Step 6：沉淀本轮经验
每轮结束必须沉淀一条可复用经验，格式尽量接近：
- 「面对做 IP 的老板时，标题里明确结果比抽象洞察更容易形成点击动作。」
- 「同一主题下，痛点切口比方法切口更容易打到冷用户。」
- 「开头先给冲突再给解释，比先铺背景更容易抓停。」

## 常见误区

### 误区 1：一上来改整篇
这会破坏可比性。先改包装层，不要一轮里全改。

### 误区 2：所有版本都差不多
这不是实验，是凑数。版本之间必须有策略差异。

### 误区 3：为了点击而牺牲信任
标题党、夸大承诺、廉价套路，不值得保留。

### 误区 4：没有提升也硬选一个
如果没有明显更好，就保留 baseline。

### 误区 5：把 Task / Sub Agent 当默认依赖
它们是增强模块，不是必选项。

## 3 个真实测试用例

### 测试 1：公众号标题优化
用户说：

> 我有一篇准备发公众号的文章，主题是「为什么大部分人做 Agent，不是死在不会写 prompt，而是死在没有闭环」。别重写全文，只帮我优化标题。目标读者是做 IP 的个人老板和一人公司创业者。原题是：《做 Agent，最难的不是 prompt》；我想要更高点开率，但不能标题党，也不能搞很重的焦虑贩卖。请按闭环方式给我 3 个标题版本，并选一个最值得发的。

你应该：
- 冻结 baseline 标题
- 只改标题变量
- 生成 3 个策略明显不同的标题
- 用 rubric 打分
- 明确保留一个新版本，或说明为什么 baseline 更稳

### 测试 2：视频号开头优化
用户说：

> 这是一条视频号口播的核心观点：「很多老板以为自己缺的是流量，实际上缺的是一个能持续试错的内容系统。」我不想重写整条口播，只想优化开头前 3 句。目标用户是已经有小业务闭环、正在做内容但播放量上不去的老板。目标是提高抓停和继续看下去的意愿，风格要真实、直接、有判断，别像平台鸡汤。请给我 3 个开头版本，主 Agent 先评测；如果你觉得两个版本很接近，可以再加一个目标用户视角的 Sub Agent 增强评审。

你应该：
- 锁定内容核心观点
- 只改开头前 3 句
- 生成 3 个明显不同的开头
- 默认主 Agent 评分
- 只有真的难分时，才追加目标用户视角增强评审

### 测试 3：CTA 优化
用户说：

> 我有一篇内容已经写完，正文主要讲「OpenClaw + Agent 工作流怎么帮一人公司提高内容产能」，现在卡在结尾 CTA 太弱。我的目标不是收割式成交，而是让对内容系统和课程感兴趣的人愿意进一步私聊我。请不要改正文，只优化结尾 CTA。目标用户是想做个人 IP、已经有服务或课程雏形的创始人。给我 3 个 CTA 版本，并明确告诉我哪一个最适合长期信任型关系。

你应该：
- 保持正文不动
- 只优化 CTA
- 重点比较行动拉力与长期信任感
- 明确拒绝过于收割式、压迫式的 CTA
- 给出清晰保留结论

## 对模型的提醒

- 这是一个闭环优化 skill，不是全文重写 skill。
- 如果问题真正卡在内容本体不足，不要硬优化包装，直接指出。
- 不要假装所有版本都差不多。必须表态：谁更好，为什么。
- 默认由主 Agent 评测，Task / Sub Agent 只是增强模块。
- 真正的最终裁判永远是未来的真实业务数据；当前 rubric 只是上线前的结构化预判。
- 这个 skill 的价值不只是输出一个更强版本，而是帮用户形成自己的包装实验方法。
